Rust中的错误处理

阅读时长:   4 分

Rust 提供了两种主要的错误处理方式：

  1. 不可恢复错误（Unrecoverable Errors）：使用 panic! 宏。
  2. 可恢复错误（Recoverable Errors）：使用 Result<T, E> 类型。

错误处理在 Rust 中至关重要，既可以提升代码的健壮性，又能提高可读性和维护性。本章将详细介绍 Rust 的错误处理机制，并给出相应的示例。

1. 不可恢复错误 (panic!)

当程序遇到严重错误无法继续运行时，Rust 提供 panic! 宏来立即终止程序，并打印错误消息和回溯信息。

1.1 panic! 的基本使用

fn panic_demo() {
    let v = vec![1, 2, 3];
    v[99]; // 访问越界，导致 panic
}

运行时会崩溃，并输出：

thread 'main' panicked at 'index out of bounds: the len is 3 but the index is 99'

1.2 panic! 的典型应用场景

  • 代码进入非法状态（如数组索引超出范围）。
  • 不希望程序继续运行，如关键文件缺失、系统错误等。
  • 用于开发调试，以便尽早发现错误（如 unreachable!() ）。

2. 可恢复错误 (Result<T, E>)

对于可以处理的错误，Rust 提供了 Result<T, E> 枚举：

enum Result<T, E> {
    Ok(T),  // 成功，返回 T
    Err(E), // 失败，返回错误信息 E
}

2.1 使用 match 处理 Result

use std::fs::File;
use std::io::ErrorKind;

fn result_demo() {
    let f = File::open("a.txt");
    let f = match f {
        Ok(file) => file,
        Err(err) => match err.kind() {
            ErrorKind::NotFound => match File::create("a.txt") {
                Ok(fc) => fc,
                Err(err1) => panic!("Failed to create a.txt {:?}", err1),
            },
            other_error => panic!("Failed to open the file {:?}", other_error),
        },
    };
}
  • File::open("a.txt") 可能成功，也可能失败。
  • 使用 match 处理 Ok 和 Err，如果文件不存在则尝试创建。
  • 若创建失败，则使用 panic! 终止程序。

2.2 使用 unwrap_or_else 简化 match

use std::io::Read;
use std::io;

fn result_demo_optimize() {
    let f = File::open("a.txt").unwrap_or_else(|error| {
        if error.kind() == ErrorKind::NotFound {
            File::create("a.txt").unwrap_or_else(|error| {
                panic!("Failed to create file {:?}", error);
            })
        } else {
            panic!("Failed to open file {:?}", error);
        }
    });
}
  • unwrap_or_else 允许更简洁地处理错误。
  • 避免了多层 match 嵌套，提高可读性。

3. 传播错误（Error Propagation）

在函数中使用 Result<T, E> 返回错误，并将错误传递给调用者。

3.1 使用 match 传播错误

fn propagating_error() -> Result<String, io::Error> {
    let f = File::open("a.txt");
    let mut f = match f {
        Ok(file) => file,
        Err(error) => return Err(error),
    };

    let mut s = String::new();
    match f.read_to_string(&mut s) {
        Ok(_) => Ok(s),
        Err(e) => Err(e),
    }
}
  • 如果 File::open("a.txt") 失败，则返回 Err。
  • 否则读取文件内容并返回 String。

3.2 使用 ? 运算符优化传播错误

fn propagating_error_opmitize() -> Result<String, io::Error> {
    let mut f = File::open("b.txt")?;
    let mut s = String::new();
    f.read_to_string(&mut s)?;
    Ok(s)
}
  • ? 语法简化了错误处理，自动传播 Err(e)。

3.3 进一步优化

fn propagating_error_opmitize_2() -> Result<String, io::Error> {
    let mut s = String::new();
    File::open("b.txt")?.read_to_string(&mut s)?;
    Ok(s)
}
  • 进一步减少了变量声明，使代码更简洁。

3.4 使用 fs::read_to_string 直接读取文件

use std::fs;

fn propagating_error_opmitize_3() -> Result<String, io::Error> {
    fs::read_to_string("a.txt")
}
  • fs::read_to_string 是最简洁的方式，直接返回 Result<String, io::Error>。

4. 选择 panic! 还是 Result？

4.1 什么时候使用 panic!

  • 代码进入非法状态（如索引越界）。
  • 程序无法继续运行，如关键数据丢失。
  • 开发过程中暴露 bug（如 unreachable!() ）。

4.2 什么时候使用 Result<T, E>

  • 错误是可恢复的，如文件未找到、网络请求失败。
  • 希望调用者决定如何处理错误。
  • 构建健壮的 API，避免 panic! 影响整个程序。

4.3 expect() 提供更明确的错误信息

fn main() {
    let f = File::open("hello.txt").expect("无法打开文件 hello.txt");
}

4.4 综述

  • panic! 适用于 严重错误，代码无法继续执行。
  • Result 适用于 可恢复错误，让调用者决定处理方式。
  • ? 语法简化了 Result 传播，使代码更简洁可读。
