1 .线程（Threads）

在大部分现代操作系统中，已执行程序的代码在一个 进程（process）中运行，操作系统则负责管理多个进程。在程序内部，也可以拥有多个同时运行的独立部分。运行这些独立部分的功能被称为 线程（threads）。

将程序中的计算拆分进多个线程可以改善性能，因为程序可以同时进行多个任务，不过这也会增加复杂性。因为线程是同时运行的，所以无法预先保证不同线程中的代码的执行顺序。这会导致诸如此类的问题：

  • 竞争状态（Race conditions），多个线程以不一致的顺序访问数据或资源
  • 死锁（Deadlocks），两个线程相互等待对方停止使用其所拥有的资源，这会阻止它们继续运行
  • 只会发生在特定情况且难以稳定重现和修复的 bug

编程语言有一些不同的方法来实现线程。很多操作系统提供了创建新线程的 API。这种由编程语言调用操作系统 API 创建线程的模型有时被称为 1:1，一个 OS 线程对应一个语言线程。

很多编程语言提供了自己特殊的线程实现。编程语言提供的线程被称为 绿色（green）线程，使用绿色线程的语言会在不同数量的 OS 线程的上下文中执行它们。为此，绿色线程模式被称为 M:N 模型：M 个绿色线程对应 N 个 OS 线程，这里 M 和 N 不必相同。

Rust 标准库只提供了 1:1 线程模型实现。由于 Rust 是较为底层的语言，如果你愿意牺牲性能来换取抽象，以获得对线程运行更精细的控制及更低的上下文切换成本，你可以使用实现了 M:N 线程模型的 crate。

1. 创建线程

Rust 提供了 std::thread 模块来创建和管理线程，使用 thread::spawn 来生成新线程。

use std::thread;
use std::time::Duration;

fn main() {
    thread::spawn(|| {
        for i in 1..5 {
            println!("thread from spawn : {}", i);
            thread::sleep(Duration::from_millis(500));
        }
    });

    for i in 1..5 {
        println!("thread from main : {}", i);
        thread::sleep(Duration::from_millis(1000));
    }
}

关键点：

  • thread::spawn 创建新线程。
  • thread::sleep 用于暂停线程。
  • 主线程和子线程并行执行。

注意这个函数编写的方式，当主线程结束时，新线程也会结束，而不管其是否执行完毕。这个程序的输出可能每次都略有不同，不过它大体上看起来像这样：

hi number 1 from the main thread!
hi number 1 from the spawned thread!
hi number 2 from the main thread!
hi number 2 from the spawned thread!
hi number 3 from the main thread!
hi number 3 from the spawned thread!
hi number 4 from the main thread!
hi number 4 from the spawned thread!
hi number 5 from the spawned thread!

2. 线程的 Join

使用 join 确保主线程等待子线程完成。

use std::thread;

fn main() {
    let handle = thread::spawn(|| {
        for i in 1..5 {
            println!("thread from spawn : {}", i);
        }
    });

    handle.join().unwrap(); // 阻塞（Blocking）等待子线程完成
    println!("thread from spawn finish");
}

主线程会等待直到新建线程执行完毕之后才开始执行 for 循环，所以输出将不会交替出现，如下所示：

hi number 1 from the spawned thread!
hi number 2 from the spawned thread!
hi number 3 from the spawned thread!
hi number 4 from the spawned thread!
hi number 5 from the spawned thread!
hi number 6 from the spawned thread!
hi number 7 from the spawned thread!
hi number 8 from the spawned thread!
hi number 9 from the spawned thread!
hi number 1 from the main thread!
hi number 2 from the main thread!
hi number 3 from the main thread!
hi number 4 from the main thread!

3.线程与 move 闭包

move 闭包，其经常与 thread::spawn 一起使用，因为它允许我们在一个线程中使用另一个线程的数据。

src/main.rs

use std::thread;

fn main() {
    let v = vec![1, 2, 3];

    let handle = thread::spawn(move || {
        println!("Here's a vector: {:?}", v);
    });

    handle.join().unwrap();
}

使用 move 关键字强制获取它使用的值的所有权

2. 消息传递（Message Passing）

一个日益流行的确保安全并发的方式是 消息传递（message passing），这里线程或 actor 通过发送包含数据的消息来相互沟通。这个思想来源于 Go 编程语言文档中 的口号：“不要通过共享内存来通讯；而是通过通讯来共享内存。”（“Do not communicate by sharing memory; instead, share memory by communicating.”）

Rust 提供了 通道（channels） 进行线程间通信，遵循 “通过消息传递而非共享状态” 原则。你可以将其想象为一个水流的通道，比如河流或小溪。如果你将诸如橡皮鸭或小船之类的东西放入其中，它们会顺流而下到达下游。

编程中的通道有两部分组成，一个发送者（transmitter）和一个接收者（receiver）。发送者位于上游位置，在这里可以将橡皮鸭放入河中，接收者则位于下游，橡皮鸭最终会漂流至此。代码中的一部分调用发送者的方法以及希望发送的数据，另一部分则检查接收端收到的消息。当发送者或接收者任一被丢弃时可以认为通道被 关闭（closed）了。

1. 创建通道

这里使用 mpsc::channel 函数创建一个新的通道；mpsc 是 多个生产者，单个消费者（multiple producer, single consumer）的缩写。简而言之，Rust 标准库实现通道的方式意味着一个通道可以有多个产生值的 发送（sending）端，但只能有一个消费这些值的 接收（receiving）端。

use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel(); // 创建通道

    thread::spawn(move || {
        let msg = String::from("Hello from thread");
        tx.send(msg).unwrap(); // 发送消息
        //println!("{}",msg); //value used here after move
    });

    let received = rx.recv().unwrap(); // 接收消息
    println!("接收到: {}", received);
}

关键点：

  • mpsc::channel 函数返回一个元组：第一个元素是发送端，而第二个元素是接收端。由于历史原因，tx 和 rx 通常作为 发送者（transmitter）和 接收者（receiver）的缩写，
  • tx.send(msg) 发送消息返回一个 Result<T, E> 类型，函数获取其参数的所有权并移动这个值归接收者所有。如果接收端已经被丢弃了，将没有发送值的目标，发送操作会返回错误。在这个例子中，出错的时候调用 unwrap 产生 panic。
  • rx.recv() 阻塞接收，rx.try_recv() 非阻塞接收。
    • recv，它是 receive 的缩写。这个方法会阻塞主线程执行直到从通道中接收一个值。一旦发送了一个值，recv 会在一个 Result<T, E> 中返回它。当通道发送端关闭，recv 会返回一个错误表明不会再有新的值到来了。
    • try_recv 不会阻塞，相反它立刻返回一个 Result<T, E>：Ok 值包含可用的信息，而 Err 值代表此时没有任何消息。如果线程在等待消息过程中还有其他工作时使用 try_recv 很有用：可以编写一个循环来频繁调用 try_recv，在有可用消息时进行处理，其余时候则处理一会其他工作直到再次检查。

运行这段代码你将看到如下输出

Receive msg: Hello form thread

2. 多个发送者

克隆 tx 使多个线程可发送消息。

use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    let (tx, rx) = mpsc::channel();
    let tx1 = tx.clone();

    thread::spawn(move ||{
        let msgs = vec![
            String::from("hi"),
            String::from("from"),
            String::from("the"),
            String::from("Rustacean"),
        ];

        for i in msgs{
            tx1.send(i).unwrap();
            thread::sleep(Duration::from_millis(200));
        }
    });

    thread::spawn(move ||{
        let msgs = vec![
            String::from("more"),
            String::from("msg"),
            String::from("for"),
            String::from("u"),
        ];

        for i in msgs{
            tx.send(i).unwrap();
            thread::sleep(Duration::from_millis(200));
        }
    });
    for received in rx{
        println!("{}",received);
    }
}

这一次，在创建新线程之前，我们对通道的发送端调用了 clone 方法。这会给我们一个可以传递给第一个新建线程的发送端句柄。我们会将原始的通道发送端传递给第二个新建线程。这样就会有两个线程，每个线程将向通道的接收端发送不同的消息。

如果运行这些代码，你 可能 会看到这样的输出：

more
from
msg
for
the
u
Rustacean

3 .共享状态（Shared State）

虽然消息传递是一个很好的处理并发的方式，但并不是唯一一个

在某种程度上，任何编程语言中的通道都类似于单所有权，因为一旦将一个值传送到通道中，将无法再使用这个值。共享内存类似于多所有权：多个线程可以同时访问相同的内存位置。然而这会增加额外的复杂性，因为需要以某种方式管理这些不同的所有者。

Rust 的类型系统和所有权规则极大的协助了正确地管理这些所有权。作为一个例子，让我们看看互斥器，一个更为常见的共享内存并发原语。

1.互斥器

互斥器（mutex）是 mutual exclusion 的缩写，也就是说，任意时刻，其只允许一个线程访问某些数据。为了访问互斥器中的数据，线程首先需要通过获取互斥器的 锁（lock）来表明其希望访问数据。锁是一个作为互斥器一部分的数据结构，它记录谁有数据的排他访问权。因此，我们描述互斥器为通过锁系统 保护（guarding）其数据。

关于互斥器请永远记住：

  1. 在使用数据之前尝试获取锁。
  2. 处理完被互斥器所保护的数据之后，必须解锁数据，这样其他线程才能够获取锁。

2. Mutex<T> 互斥锁

Rust 使用 Arc<Mutex<T>> 进行共享状态并发。

use std::sync::Mutex;

fn main() {
    let m = Mutex::new(5);
    {
        let mut num = m.lock().unwrap(); // 获取锁 这个调用会阻塞当前线程，直到我们拥有锁为止
        *num = 10;
    } // 作用域结束时自动释放锁
    println!("m = {:?}", m);
}

关键点：

  • Mutex::new(value) 创建互斥锁。
  • Mutex<T>是一个智能指针。更准确的说，lock 调用 返回 一个叫做 MutexGuard 的智能指针, 这个智能指针实现了 Deref 来指向其内部数据
  • lock().unwrap() 获取锁。
  • 作用域结束时自动释放锁。Mutex<T>这个智能指针提供了一个 Drop 实现当 MutexGuard 离开作用域时自动释放锁

运行结果：

mutex_lock is: Mutex { data: 6, poisoned: false, .. }

3. 在线程间共享 Mutex

假设我们需要写一个计数器，启动10个线程，每个线程将一个共享计数器加1，最终打印计数器的值（预期为10）。

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
   // 创建一个共享的计数器，初始值为 0
    // Arc（原子引用计数）允许多线程共享所有权，Mutex 提供线程安全的互斥锁,Rc在这里不安全
    let counter = Arc::new(Mutex::new(0));

    // 创建一个向量，用于存储线程句柄，以便后续等待线程完成
    let mut handles = vec![];

    // 启动 10 个线程，每个线程将计数器加 1
    for _ in 0..10 { // 使用 _ 因为循环变量未使用
        // 克隆 counter 的 Arc 引用，增加引用计数，为新线程准备一个独立的共享引用
        let counter = Arc::clone(&counter);

        // 启动一个新线程，使用 move 闭包将 counter 的所有权移入线程
        let handle = thread::spawn(move || {
            // 获取 Mutex 的锁，lock() 返回一个 MutexGuard，unwrap() 处理可能的错误
            // Mutex 确保同一时间只有一个线程可以修改 counter 的值
            let mut num = counter.lock().unwrap();

            // 通过解引用 (*) 修改计数器的值，加 1
            *num += 1;
            // num 离开作用域时，锁自动释放（MutexGuard 的 Drop 实现）
        });

        // 将线程句柄存入向量，以便后续等待
        handles.push(handle);
    }

    // 等待所有线程完成
    // 遍历句柄向量，调用 join() 阻塞主线程直到子线程结束，unwrap() 处理可能的错误
    for handle in handles {
        handle.join().unwrap();
    }

    // 打印最终结果
    // 主线程获取 counter 的锁，访问并打印其值，* 解引用 MutexGuard 获取内部 i32
    println!("Result is: {}", *counter.lock().unwrap());
}

关键点：

  • Arc<Mutex<T>> 原子引用计数，适合多线程共享。
  • Arc::clone(&counter) 克隆 Arc，增加引用计数，引用全释放时内存才清理

4. Sync 和 Send 特征

在 Rust 中，Send 和 Sync 是 标记（marker）特征，用于表示 线程安全的类型。它们的定义如下：

1. Send（允许跨线程传递所有权）

#[lang = "send"]
pub unsafe auto trait Send {}
  • 含义：Send 表示该类型的所有权可以安全地在线程间移动（通过 thread::spawn() 传递）。
  • 默认实现：Rust 自动为大多数类型实现 Send，除非它们包含了非 Send 类型。

常见 Send 类型：

  • 基础类型（i32、f64、char、bool）。
  • Box<T>、Vec<T>（只要 T 也是 Send）。
  • Arc<T>（不是 Rc<T>，因为 Rc<T> 不是 Send）。
  • Mutex<T>（只要 T 也是 Send）。

非 Send 类型（无法安全跨线程传递）：

  • Rc<T>（不是 Send，因为 Rc 的引用计数不是原子操作）。
  • *const T / *mut T（裸指针可能导致数据竞争）。
  • RefCell<T>（RefCell 运行时借用检查，不适用于多线程）。

默认情况下，Rust 允许线程安全的类型实现 Send，但 Rc<T> 不是 Send，因此不能在线程间传递：

use std::rc::Rc;
use std::thread;

fn main() {
    let rc = Rc::new(5);

    thread::spawn(move || {
        println!("值: {}", rc);
    }).join().unwrap(); // ❌ 编译错误：`Rc<T>` 不能跨线程传递
}

原因：
Rc<T> 依赖引用计数（ref count），但它不是线程安全的。如果多个线程同时修改 Rc<T> 的引用计数，可能导致 数据竞争。

解决方案：使用 Arc<T>（原子引用计数）

use std::sync::Arc;
use std::thread;

fn main() {
    let data = Arc::new(5);
    let data1 = Arc::clone(&data);

    thread::spawn(move || {
        println!("线程读取值: {}", data1);
    }).join().unwrap();
}

✅ Arc<T> 通过原子操作确保线程安全，因此可以安全地在多个线程之间共享。

2. Sync（允许多线程共享 &T）

#[lang = "sync"]
pub unsafe auto trait Sync {}
  • 含义：如果 T: Sync，则 &T 可以安全地被多个线程共享，即 &T 的多个 不可变引用 可以在不同线程中使用。
  • 默认实现：Rust 自动为大多数类型实现 Sync，除非它们内部包含了非 Sync 类型。

常见 Sync 类型：

  • 基础类型（i32、f64、char）。
  • &T 只要 T 也是 Sync。
  • Arc<T>（只要 T 也是 Sync）。
  • Mutex<T>、RwLock<T>（用于保证内部数据的并发访问安全）。

非 Sync 类型（无法多线程共享 &T）：

  • Rc<T>（不是 Sync，因为它的引用计数不是原子操作）。
  • Cell<T> / RefCell<T>（不是 Sync，因为它们是 !Send 且允许运行时可变借用）。
  • Mutex<T> 不是 Sync 如果 T 不是 Send，因为锁的所有权不能安全地在多个线程间共享。

如果 T 是 Sync，则 &T（不可变引用）可以在线程间安全共享。

use std::sync::Arc;
use std::thread;

fn main() {
    let data = Arc::new(5);
    let data1 = Arc::clone(&data);

    let handle = thread::spawn(move || {
        println!("线程读取值: {}", data1);
    });

    handle.join().unwrap();
}

在这个示例中，Arc<T> 允许多个线程共享数据，并且保证安全。

3. Send 与 Sync 的区别

特征 作用 适用于
Send 允许数据的 所有权 在线程间移动 适用于 跨线程传递 数据
Sync 允许数据的 引用 在线程间共享 适用于 多线程共享 不可变数据

原则：

  • 大多数 Rust 类型默认是 Send 和 Sync，除了 Rc<T>（因为它不是线程安全的）。
  • 如果一个类型包含 Rc<T>，那么它也不是 Send，但如果改用 Arc<T>，就可以变成 Send。

4. 总结

  • Send：数据可以在线程间 传递（如 Arc<T>）。
  • Sync：数据的 引用 可以安全地在多个线程共享。
  • Rc<T> 不是 Send，但 Arc<T> 是 Send + Sync，适用于并发情况。
